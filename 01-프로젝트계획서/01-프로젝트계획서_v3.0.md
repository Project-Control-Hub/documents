# Project Control Hub 프로젝트 계획서

## 1. 프로젝트 개요

### 1.1 배경 및 목적
- **배경**: 소프트웨어 개발팀의 애자일(Agile) 방법론 지원에 특화된 이슈 트래킹 및 프로젝트 관리 도구가 필요함. 팀 워크플로우 설계, 릴리즈 관리, 자동화, 보고까지 통합하는 협업 플랫폼 구축
- **목적**: Project Control Hub를 구축하여 이슈 관리, 스프린트 운영, 릴리즈 관리, 대시보드 등 통합 협업 환경 제공

### 1.2 프로젝트 범위

> FR 코드는 [07-요구사항정의서](../07-요구사항정의서/07-요구사항정의서_v2.2.md) 순번 체계(FR-001~033, FR-MOBILE-001~004) 기준입니다. 각 FR의 상세 Acceptance Criteria, 관련 화면, 관련 API는 요구사항정의서를 참조하십시오.

#### In-Scope (포함 범위)

| 기능 그룹 | 포함 기능 | 구현 수준 | 관련 FR |
|-----------|-----------|-----------|---------|
| 이슈 관리 | 이슈 CRUD (Epic/Story/Task/Bug/Sub-task), 이슈 타입별 생성, 이슈 상태 관리 (6단계), 우선순위 설정, 담당자 지정, 레이블 & 컴포넌트, 이슈 간 링크 (Blocks/Duplicates/Relates to) | 전체 구현 | FR-001~007 |
| 프로젝트 관리 | 스크럼 보드, 칸반 보드 + WIP 제한, 백로그 관리, 스프린트 생성/시작/완료, 로드맵 (Epic 타임라인) | 전체 구현 | FR-008~012 |

## 6. Definition of Ready / Definition of Done

> 제품 기능 정의서 (`Project_Control_Hub.md`) 섹션 10 기준. Sprint Planning 및 이슈 완료 판단의 공식 기준으로 사용.

### 6.1 Definition of Ready (DoR)

Sprint Planning에서 이슈를 Sprint에 포함하기 위한 최소 준비 조건입니다.

| 항목 | 확인 내용 |
|------|-----------|
| 요구사항 명확 | Acceptance Criteria(완료 조건)가 명문화되어 있음 |
| UI/UX 산출물 | 관련 화면 설계 또는 와이어프레임이 존재 |
| API 스펙 정의 | 연관 API 엔드포인트 및 요청/응답 형식 확정 |
| Story Point 산정 | Planning Poker 등을 통해 SP가 합의됨 |
| 의존성 파악 | 선행 이슈(Blocked by)가 완료되었거나 계획 확정 |
| 이슈 크기 적정 | 단일 Sprint 내 완료 가능한 규모 (SP 8 이하 권장) |

### 6.2 Definition of Done (DoD)

이슈를 Done 상태로 전환하기 위한 필수 완료 조건입니다.

| 항목 | 확인 내용 |
|------|-----------|
| 코드 구현 완료 | Acceptance Criteria 전체 충족 |
| 코드 리뷰 완료 | 최소 1인 이상 Reviewer 승인(Approve) |
| 단위 테스트 통과 | 신규 코드 커버리지 80% 이상 |
| QA 테스트 통과 | 기능 테스트 시나리오 전체 Pass |
| 문서 업데이트 | API 명세서, README 등 관련 문서 반영 |
| 배포 가능 상태 | main/develop 브랜치 머지 완료, 빌드 성공 |
| 회귀 테스트 확인 | 기존 기능에 영향 없음 확인 |

## 8. 변경 관리 절차

> 프로젝트 범위, 일정, 요구사항 변경 발생 시 아래 절차를 준수합니다.

### 8.1 변경 관리 프로세스

| 단계 | 활동 | 담당 | 산출물 |
|------|------|------|--------|
| 1. 변경 요청 (CR) 접수 | 변경 사항 내용, 사유, 긴급도 작성 후 PM에 제출 | 요청자 (누구나) | 변경 요청서 (CR Form) |
| 2. 영향 분석 | 일정, 범위, 비용, 기술적 영향 분석 | PM + 백엔드/프론트엔드 리드 | 영향 분석서 |
| 3. 승인 | 영향 분석 결과 검토 후 승인 또는 반려 결정 | PM (소규모) / 이해관계자 포함 (중대규모) | 승인 기록 (회의록) |
| 4. 백로그 반영 | 승인된 변경을 이슈로 등록, 스프린트 배정 | PM + 개발팀 | 이슈, 업데이트된 백로그 |
| 5. 구현 및 검증 | 변경 사항 개발 후 DoD 기준 검증, QA 수행 | 개발팀 + QA | 테스트 결과, 배포 산출물 |
| 6. 완료 확인 | 변경 적용 결과 확인 및 문서 업데이트 | PM | 업데이트된 계획서, 이력 기록 |

### 8.2 변경 규모 분류 기준

| 분류 | 기준 | 승인 권한 |
|------|------|-----------|
| 소규모 | SP 합계 5 이하, 일정 영향 없음 | PM 단독 승인 |
| 중규모 | SP 합계 6~13 또는 일정 1주 이내 영향 | PM + 팀 리더 승인 |
| 대규모 | SP 합계 14 이상 또는 일정 2주 이상 영향, 범위 변경 | PM + 이해관계자 전체 합의 |

| 스프린트 리뷰 | 격주 금요일 오후 | 1~2시간 | 전체 팀 + 이해관계자 | 완성 기능 데모, 이해관계자 피드백 |
| 회고 (Retrospective) | 격주 금요일 말 | 1시간 | 전체 팀 | Keep / Problem / Try 도출 |
| 주간 PM 보고 | 매주 | 30분 | PM + 이해관계자 | 진척 현황, 리스크, 의사결정 필요 사항 |

### 9.3 에스컬레이션 절차

| 단계 | 조건 | 에스컬레이션 대상 | 목표 해결 시간 |
|------|------|------------------|---------------|
| Level 1 | 블로커 발생 (개인 해결 불가) | 팀 리더 / PM | Daily Scrum 당일 |
| Level 2 | 일정 1주 이상 지연 예상, 범위 변경 필요 | PM + 이해관계자 | 48시간 이내 |
| Level 3 | 예산 초과, 대규모 범위 변경, 팀 이탈 | 스폰서 / 경영진 | 24시간 이내 |

### 9.4 의사결정 프로세스

- **기술 결정**: 관련 개발자 합의 후 팀 리더 최종 확인. 이견 시 PM 중재.
- **범위 변경**: 변경 관리 절차(섹션 8) 준수.
- **우선순위 변경**: PM이 Product Owner 역할로 결정, Sprint 중 변경 시 팀 합의 필요.
- **회의 의사결정**: 회의록 작성 후 참여자 확인, 이슈 또는 문서로 기록 보존.

## 11. 버전 명명 규칙

Semantic Versioning(SemVer) 기준을 따릅니다. (제품 기능 정의서 (`Project_Control_Hub.md`) 섹션 12.3 기준)

```
형식: MAJOR.MINOR.PATCH

MAJOR : 하위 호환 불가 변경 (대규모 리팩토링, API Breaking Change)
MINOR : 하위 호환 신규 기능 추가
PATCH : 버그 수정, 마이너 개선

예시
  v1.0.0  → 최초 릴리즈
  v1.1.0  → 신규 기능 추가
  v1.1.1  → 긴급 버그 수정
```

문서 버전도 동일한 규칙을 적용합니다.
