## 1. Critical 이슈 (즉시 해결 필요)

프로젝트 진행을 차단하거나 문서 간 추적이 불가능한 이슈입니다.

### 1.1 FR 코드 체계 이중 사용 (3개 문서 영향)

| 항목 | 상세 |
|------|------|
| **관련 문서** | 00-스케줄, 01-프로젝트계획서, 07-요구사항정의서 |
| **현상** | 프로젝트계획서/스케줄은 모듈 접두사 체계(FR-ISSUE-001, FR-WF-001 등 ~85개), 요구사항정의서는 순번 체계(FR-001~033 + FR-MOBILE-001~004 = 37개) 사용 |
| **영향** | 요구사항-스케줄-계획서 간 추적 불가능. 동일 기능에 서로 다른 ID가 할당됨 |
| **권장** | 하나의 체계로 통일하고, 3개 문서에 일관 적용. 매핑 테이블 작성 필수 |

### 1.2 ERD-API 정합성 갭 (v3.0 신규 테이블 6개 API 부재)

| 항목 | 상세 |
|------|------|
| **관련 문서** | 02-ERD v3.0, 04-API정의서 v3.0 |
| **현상** | ERD에 추가된 PLANNING_POKER_SESSION/VOTE, SCREEN_SCHEME/FIELD, WIP_LIMIT, SECURITY_LEVEL_PERMISSION 등 6개 테이블의 API 엔드포인트가 0개 |
| **영향** | 데이터 모델은 확장되었으나 API 계층이 미반영. 구현 시 혼란 발생 |
| **권장** | 사용자 대면 기능(WIP Limit, Planning Poker)은 API 추가 필수. 내부 전용이면 명시 |

### 1.3 SCR 코드 매핑 불일치 (UI/UX 3개 문서 영향)

| 항목 | 상세 |
|------|------|
| **관련 문서** | 05-화면흐름시퀀스, 06-화면기능정의서, 09-스토리보드 |
| **현상** | 05-시퀀스: SCR-005="이슈 상세", SCR-006="이슈 생성". 06-화면기능정의서: SCR-005="칸반 보드", SCR-006="이슈 상세", SCR-007="이슈 생성" |
| **영향** | 동일 화면에 서로 다른 SCR 코드가 할당되어 구현 시 참조 혼란 불가피 |
| **권장** | 06-화면기능정의서 기준으로 통일 후 05/09 문서 수정 |

### 1.4 BOARD 엔티티 부재

| 항목 | 상세 |
|------|------|
| **관련 문서** | 02-ERD, 04-API정의서 |
| **현상** | API에서 boardId를 반환하지만 ERD에 BOARD 테이블 없음. PROJECT.board_type으로 대체한 것으로 보이나, 하나의 프로젝트에 여러 보드 불가 |
| **권장** | BOARD를 별도 엔티티로 분리하거나 boardId = projectId 별칭임을 명시 |

### 1.5 Hotfix 브랜치 보호 규칙 충돌

| 항목 | 상세 |
|------|------|
| **관련 문서** | 08-Git규칙정의서, 11-코드리뷰규칙 |
| **현상** | main 브랜치 보호(Required Reviews: 2명) vs 긴급 리뷰(1명). Hotfix가 main에 직접 머지되는데 보호 규칙 바이패스 허용 여부 미정의 |
| **권장** | 바이패스 정책과 권한자를 명시적으로 추가 |

### 3.2 DevOps E2E 파이프라인 연결성

**양호**: feature 브랜치(08) → PR(08) → 코드리뷰(11) → 테스트 게이트(10) → 배포(12)의 E2E 흐름이 4개 문서에 걸쳐 일관적으로 연결됨.

**양호**: Hotfix 프로세스가 08/11/12에서 큰 틀이 일관적 (보호 규칙 충돌 1건 제외).

**양호**: 태그/버전 3자 매핑(Git Tag = Atlassian Jira Fix Version = Docker Image Tag) 규칙 존재 (Docker sha 접미사 이슈 제외).

## 7. 결론

Phase 1~3 보강을 통해 전체 문서 완성도가 **38% → 92%**로 대폭 향상되었습니다. v1.0에서 지적된 핵심 이슈(누락 테이블, API, 화면 상세, TC 확장, DevOps 자동화 등)는 대부분 해결되었습니다.

현재 잔여 이슈는 크게 3가지 유형입니다:

1. **정합성 이슈** (가장 시급): FR 코드 체계, SCR 코드, Hotfix 보호 규칙 등 문서 간 불일치
2. **v3.0 확장 갭**: ERD에 추가된 신규 테이블의 API 미커버
3. **모바일 커버리지**: Flutter 앱 관련 테스트/코딩 표준/아키텍처 상세 부족

Phase 4(정합성 통일)를 우선 진행하면 문서 간 추적성이 확보되어 구현 단계 진입이 가능합니다.
