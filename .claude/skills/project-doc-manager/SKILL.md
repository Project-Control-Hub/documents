# Project Document Manager

프로젝트에 필요한 16종 표준 문서의 폴더 구조를 생성하고, 각 문서의 기본 템플릿을 제공하며, 파일명 기반 버전 관리를 수행하는 스킬이다. 참조 문서(기능정의서, 기획서 등)를 입력받아 실무 수준의 문서를 자동 생성하고, 문서 품질 분석 및 보강 워크플로우를 지원한다.

## 명령어

이 스킬은 자연어와 명령어 스타일 모두 지원한다.

### 1. 전체 초기화: `doc init`

16개 문서 폴더와 각 폴더에 기본 템플릿을 생성한다.

**자연어 예시**: "프로젝트 문서 초기화해줘", "문서 폴더 구조 잡아줘", "프로젝트 문서 셋업"

**옵션**:
- `--from <파일경로>`: 참조 문서를 기반으로 실제 내용이 채워진 문서 생성
- `--depth <skeleton|standard|comprehensive>`: 템플릿 깊이 선택 (기본: standard)
- `--path <경로>`: 출력 경로 지정 (기본: `docs/`, 사용자가 "루트에" 등 경로를 지정하면 해당 경로 사용)
- `--profile <web|mobile|data|microservice>`: 프로젝트 타입별 문서 구성

**동작**:
1. 출력 경로 확인 (기본: `docs/`, `--path`로 변경 가능)
2. 프로필에 따라 문서 구성 결정
3. 16개 하위 폴더 생성
4. `--from`이 있으면 참조 문서를 읽고 내용 반영, 없으면 템플릿 생성
5. `--depth`에 따라 채우는 깊이 결정
6. `README.md` (문서 인덱스) 생성

**실행 방법**: `references/templates.md`를 읽고 각 문서 유형별 템플릿을 사용하여 파일을 생성한다.

### 2. 개별 문서 생성: `doc create <문서번호|문서명>`

특정 문서만 생성한다.

**자연어 예시**: "ERD 문서 만들어줘", "API 정의서 템플릿 생성해줘"

**동작**:
1. 해당 문서의 폴더가 없으면 생성
2. 기본 템플릿 파일을 v1.0으로 생성
3. 이미 존재하면 사용자에게 알림
4. README.md 인덱스 자동 갱신

### 3. 버전 업: `doc version <문서번호|문서명>`

기존 문서의 새 버전을 생성한다.

**자연어 예시**: "ERD 문서 버전 올려줘", "API 정의서 v2 만들어줘"

**동작**:
1. 해당 폴더에서 가장 최신 버전 파일을 찾음
2. 최신 버전을 복사하여 버전 번호를 +1.0 증가
3. 새 파일의 메타데이터(버전, 수정일) 업데이트
4. 변경 이력 섹션에 새 항목 추가
5. **README.md 인덱스 자동 갱신** (최신 버전 링크 업데이트)

**버전 네이밍 규칙**:
- 파일명: `{번호}-{문서명}_v{Major}.{Minor}.md`
- Major: 큰 구조 변경 시 증가 (v1.0 → v2.0)
- Minor: 내용 수정 시 증가 (v1.0 → v1.1)

### 4. 문서 목록 확인: `doc list`

현재 생성된 문서 목록과 각 문서의 최신 버전을 표시한다.

**자연어 예시**: "문서 현황 보여줘", "어떤 문서가 있어?"

### 5. 인덱스 갱신: `doc index`

`README.md`의 문서 인덱스를 현재 폴더 상태에 맞게 갱신한다.

### 6. 문서 분석: `doc analyze [문서]`

전체 또는 개별 문서의 품질을 분석한다.

**자연어 예시**: "문서 품질 분석해줘", "누락된 내용 알려줘", "문서 현황 점검해줘"

**동작**:
1. 참조 문서가 있으면 대조 분석 (없으면 업계 표준 기준)
2. 4가지 관점으로 분석:
   - **누락된 내용**: 참조 문서에 있지만 문서에 미반영된 항목
   - **깊이 부족**: 작성되었지만 플레이스홀더/예시 수준에 머문 항목
   - **문서 간 정합성**: Cross-reference 누락, 문서 간 내용 불일치
   - **업계 표준 대비**: PMBOK, Agile/Scrum, DevSecOps 등 표준 관점 보강점
3. 문서별 완성도 % 산출
4. 보강 우선순위 로드맵 생성 (Phase 1/2/3)

**출력**: 분석 레포트 Markdown 파일

**병렬 실행 권장**: 문서 그룹별로 Agent를 나누어 병렬 분석하면 빠르다.
- Team 1: 기획/요구사항 (00, 01, 07)
- Team 2: 아키텍처/데이터 (02, 03, 04)
- Team 3: UI/UX (05, 06, 09)
- Team 4: DevOps/프로세스 (08, 10, 11, 12, 13)
- Team 5: 운영/사용자 (14, 15)

### 7. 문서 보강: `doc improve [문서]`

분석 결과를 기반으로 문서를 자동 보강한다.

**자연어 예시**: "문서 보강해줘", "요구사항정의서 개선해줘", "Phase 1 보강 진행해줘"

**동작**:
1. `doc analyze` 결과 확인 (없으면 먼저 실행)
2. Phase별 보강 작업 계획 수립
3. 새 버전 생성 (v1.0 → v2.0)하며 보강 내용 반영
4. README.md 자동 갱신

**의사결정 필요 항목**: 보강 전 사용자에게 Open Questions를 소크라테스식 문답법으로 제시하고 확인받는다.

**소크라테스식 문답법 (의사결정 프로세스)**:
문서 검수/보강 시 의사결정이 필요한 항목은 다음 절차를 따른다:
1. **현상 질문**: "현재 A와 B가 불일치합니다. 이 차이가 의도된 것인지요?" — 사용자가 문제를 인식하게 한다
2. **영향 질문**: "이 불일치가 해소되지 않으면 어떤 문제가 발생할까요?" — 사용자가 영향 범위를 스스로 파악하게 한다
3. **선택지 제시**: "해결 방안으로 X와 Y가 있는데, 각각의 트레이드오프는..." — 선택지와 근거를 제시한다
4. **유도 질문**: "프로젝트의 [특성/제약]을 고려하면 어느 쪽이 더 적합할까요?" — 사용자 맥락에 맞는 결론으로 유도한다
5. **확인**: 사용자의 결정을 확인하고, 결정 근거를 문서에 기록한다

이 방법은 단순 "A or B?" 나열보다 사용자가 맥락을 이해하고 납득할 수 있는 결정을 내리게 돕는다. 항목이 많을 경우 관련 항목끼리 묶어 대화 흐름을 자연스럽게 구성한다.

**병렬 실행 권장**: 독립적인 문서들은 Agent로 병렬 보강이 가능하다.

### 8. Cross-reference: `doc crossref`

문서 간 참조 링크를 자동 삽입하고 정합성을 검증한다.

**자연어 예시**: "문서 간 연결 정리해줘", "ERD와 API 매핑 확인해줘"

**동작**:
1. ERD 엔티티 ↔ API 엔드포인트 매핑 검증
2. 요구사항 FR ↔ 화면 ID ↔ API ↔ 테스트 케이스 추적 매트릭스(RTM) 갱신
3. 아키텍처 컴포넌트 ↔ ERD/API 정합성 확인
4. 누락된 참조 링크 자동 삽입
5. 불일치 항목 리포트

## 템플릿 깊이 계층

| 깊이 | 설명 | 사용 시기 |
|------|------|----------|
| `skeleton` | 섹션 제목 + 빈 테이블/플레이스홀더 | 빠른 구조 잡기, 직접 채울 때 |
| `standard` (기본) | 예시 데이터 2~3개 포함 + 작성 가이드 코멘트 | 일반적인 프로젝트 시작 |
| `comprehensive` | 참조 문서 기반 실무 수준 (상세 조건, 전체 매핑) | 참조 문서가 있을 때, `--from` 사용 시 자동 적용 |

`--from` 옵션 사용 시 자동으로 `comprehensive` 깊이가 적용된다.

## 프로젝트 프로필

| 프로필 | 포함 문서 | 특이사항 |
|--------|----------|---------|
| `web` (기본) | 16종 전체 | 스크럼/칸반 보드 중심 |
| `mobile` | 16종 + 앱스토어 배포 가이드 | 모바일 특화 화면 흐름 |
| `data` | 00, 01, 02, 03, 04, 07, 08, 10, 11, 12, 13, 15 | 화면기능정의서(06), 스토리보드(09), 사용자매뉴얼(14) 제외, ETL/스케줄링 추가 |
| `microservice` | 16종 + 서비스 카탈로그, 이벤트 스키마 | MSA 특화 아키텍처 |

사용자가 프로필에 없는 커스텀 구성을 요청하면 유연하게 대응한다.

## 문서 포맷

모든 문서는 **Markdown + Mermaid** 혼합 포맷을 사용한다:
- 텍스트, 테이블, 체크리스트: Markdown
- 다이어그램 (ERD, 시퀀스, 플로우차트, 아키텍처): Mermaid 코드 블록

## 문서 메타데이터

모든 문서 상단에 다음 메타데이터를 포함한다:

```markdown
---
문서명: {문서 제목}
버전: v{Major}.{Minor}
작성일: {YYYY-MM-DD}
최종수정일: {YYYY-MM-DD}
작성자: {작성자}
상태: 초안 | 검토중 | 승인됨
---
```

## 변경 이력

모든 문서 하단에 변경 이력 테이블을 포함한다:

```markdown
## 변경 이력

| 버전 | 날짜 | 작성자 | 변경 내용 |
|------|------|--------|-----------|
| v1.0 | {YYYY-MM-DD} | {작성자} | 최초 작성 |
```

## 실행 가이드

### 전체 초기화 시 (기본)

1. `references/templates.md`를 읽어 각 문서 유형의 템플릿 내용을 확인
2. 출력 경로를 확인 (사용자가 지정하지 않으면 `docs/`)
3. 폴더와 파일을 생성
4. 각 템플릿의 `{YYYY-MM-DD}` 플레이스홀더를 오늘 날짜로 치환
5. 각 템플릿의 `{프로젝트명}` 플레이스홀더를 프로젝트명으로 치환 (사용자에게 확인)
6. `README.md` 인덱스 파일 생성

### 참조 문서 기반 초기화 시 (`--from`)

참조 문서를 기반으로 실제 내용이 채워진 문서를 생성하는 것이 이 스킬의 핵심 가치다. 빈 플레이스홀더가 아닌, 프로젝트의 실제 맥락이 반영된 문서를 만든다.

1. 참조 문서를 Read하여 프로젝트 컨텍스트를 추출한다
   - 프로젝트명, 기능 목록, 기술 스택, 팀 구성, 일정 등
2. 각 문서 템플릿에 참조 문서의 내용을 매핑한다
   - 기능 목록 → 07-요구사항정의서 FR 항목
   - 데이터 모델 → 02-ERD 테이블
   - API 엔드포인트 → 04-API정의서
   - 워크플로우 → 05-화면흐름시퀀스
   - 등등
3. `comprehensive` 깊이로 각 문서를 생성한다
4. **병렬 Agent 활용**: 16종 문서를 4~5개 그룹으로 나누어 동시 생성하면 빠르다

### 문서 분석 시 (`doc analyze`)

문서 품질 분석은 참조 문서와 현재 문서를 비교하여 갭을 찾는 작업이다.

1. 원본 참조 문서(있는 경우)와 현재 문서를 비교한다
2. 4가지 관점으로 분석한다:
   - 누락된 내용 (참조 문서에 있지만 현재 문서에 없는 항목)
   - 깊이 부족 (플레이스홀더/예시 수준에 머문 항목)
   - 문서 간 정합성 (ERD 엔티티 vs API 커버리지, 요구사항 vs 화면 매핑 등)
   - 업계 표준 대비 (PMBOK, Agile, DevSecOps 관점)
3. 문서별 완성도 %를 산출한다
4. 보강 우선순위 로드맵을 Phase 1(필수)/Phase 2(중요)/Phase 3(고도화)로 생성한다
5. 분석 레포트를 Markdown 파일로 저장한다

**병렬 분석 권장**: Agent 4팀으로 분할하면 효율적이다.

### 문서 보강 시 (`doc improve`)

분석 결과를 기반으로 문서 내용을 보강하고 새 버전을 생성하는 작업이다.

1. `doc analyze` 결과를 확인한다 (없으면 먼저 실행)
2. 사용자에게 Open Questions를 제시하고 의사결정을 받는다
3. Phase별 보강 작업을 계획한다
4. 각 문서의 새 버전(v2.0)을 생성하며 보강 내용을 반영한다
5. 모든 버전업 후 README.md를 자동 갱신한다

**병렬 보강 권장**: 독립적인 문서들은 Agent로 동시에 보강할 수 있다.
- 의존성 없는 문서: 01, 07, 08 등은 병렬 가능
- 의존성 있는 문서: 02(ERD) → 04(API)는 순차 진행

### 버전 업 시

1. 해당 문서 폴더에서 `ls`로 기존 파일 확인
2. 가장 높은 버전 번호의 파일을 복사
3. 버전 번호를 +1.0 (Major) 또는 +0.1 (Minor) 증가
4. 메타데이터의 버전, 최종수정일 갱신
5. 변경 이력 테이블에 새 행 추가
6. README.md 인덱스의 해당 문서 링크와 버전 업데이트

## 주의사항

- 이미 존재하는 파일을 덮어쓰지 않는다. 항상 사용자에게 확인 후 진행한다.
- 프로젝트명, 작성자 등 플레이스홀더는 첫 초기화 시 사용자에게 확인한다.
- Mermaid 다이어그램은 문서 특성에 맞는 유형을 사용한다 (ERD → erDiagram, 시퀀스 → sequenceDiagram, 아키텍처 → flowchart 등).
- `--from` 사용 시 참조 문서는 읽기만 하고 수정하지 않는다.
- `doc improve` 실행 전 의사결정이 필요한 항목(Open Questions)을 사용자에게 먼저 제시한다.
- 병렬 Agent 실행 시 같은 파일을 동시에 수정하지 않도록 주의한다.
- 버전 업 시 README.md 인덱스를 항상 함께 갱신한다.
