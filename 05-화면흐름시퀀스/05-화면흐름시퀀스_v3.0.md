# Project Control Hub 화면 흐름 시퀀스 다이어그램

## 1. 전체 화면 흐름도

```mermaid
flowchart LR
    LOGIN[로그인\nSCR-001] --> MAIN[메인 대시보드\nSCR-002]
    MAIN --> BACKLOG[백로그\nSCR-003]
    MAIN --> SCRUM[스크럼 보드\nSCR-004]
    MAIN --> KANBAN[칸반 보드\nSCR-005]
    MAIN --> SETTINGS[프로젝트 설정\nSCR-013]

    BACKLOG --> ISSUE_CREATE[이슈 생성/수정\nSCR-007]
    BACKLOG --> ISSUE_DETAIL[이슈 상세\nSCR-006]
    SCRUM --> ISSUE_DETAIL
    KANBAN --> ISSUE_DETAIL
    ISSUE_DETAIL --> COMMENT[댓글/첨부]
    ISSUE_DETAIL --> TRANSITION[상태 전환]

    MAIN --> SPRINT_MGMT[스프린트 관리\nSCR-010]
    SPRINT_MGMT --> SPRINT_PLANNING[스프린트 플래닝]
    SPRINT_MGMT --> SPRINT_REVIEW[스프린트 리뷰]
    SPRINT_MGMT --> POKER_MODAL[Planning Poker 모달\nSCR-010 내]

    MAIN --> RELEASE[릴리즈/버전 관리\nSCR-011]
    MAIN --> REPORTS[리포트\nSCR-012]
    MAIN --> SEARCH[JQL 검색\nSCR-008]
    MAIN --> AUDIT[Audit Log\nSCR-014]
    MAIN --> DASHBOARD[대시보드 커스터마이징\nSCR-009]

    SETTINGS --> WORKFLOW[워크플로우 설정]
    SETTINGS --> PERMISSION[권한 관리]
    SETTINGS --> AUTOMATION_SET[자동화 규칙]
    SETTINGS --> LOGOUT[로그아웃]
    LOGOUT --> LOGIN
```

## 2. 주요 시퀀스 다이어그램

### 2.1 로그인 흐름

```mermaid
sequenceDiagram
    actor U as 사용자
    participant V as 화면
    participant A as API Server
    participant D as Database
    participant Cache as Token Cache

    U->>V: 로그인 페이지 접근
    V-->>U: 로그인 폼 표시
    U->>V: 이메일/비밀번호 입력
    V->>A: POST /auth/login
    A->>D: 사용자 인증 확인
    alt 인증 성공
        D-->>A: 사용자 정보 + 역할(RBAC)
        A-->>V: JWT Token (Access + Refresh)
        V-->>U: 메인 대시보드로 이동
    else 인증 실패 (5회 초과)
        D-->>A: 실패
        A-->>V: 계정 잠금 에러
        V-->>U: "계정이 잠겼습니다 (30분)" 메시지
    else 인증 실패
        D-->>A: 실패
        A-->>V: 에러 응답
        V-->>U: 에러 메시지 표시
    end

    Note over U,Cache: Token Refresh 흐름
    U->>V: 보호된 리소스 요청
    V->>A: API 요청 (Access Token 포함)
    A-->>V: 401 Unauthorized (Access Token 만료)
    V->>A: POST /auth/refresh (Refresh Token 전송)
    alt Refresh Token 유효
        A->>Cache: 새 Access Token 발급
        Cache-->>A: 저장 완료
        A-->>V: 새 Access Token 반환
        V->>A: 원래 API 요청 재시도
        A-->>V: 정상 응답
        V-->>U: 화면 정상 표시
    else Refresh Token 만료/무효
        A-->>V: 401 Refresh Token Invalid
        V-->>U: 로그인 화면으로 리다이렉트
    end
```

### 2.2 이슈 생성 및 스프린트 배정 흐름

```mermaid
sequenceDiagram
    actor PO as Product Owner
    participant BL as 백로그 화면(SCR-003)
    participant A as API Server
    participant D as Database
    participant WF as 워크플로우 엔진

    PO->>BL: 이슈 생성 버튼 클릭
    BL-->>PO: 이슈 생성 폼 표시 (SCR-007)
    PO->>BL: 이슈 정보 입력<br/>(타입, 제목, 설명, 우선순위, SP)

    alt 이슈 타입 = Bug
        BL->>BL: 유효성 검사<br/>(재현절차 필수 확인)
        alt 재현절차 미입력
            BL-->>PO: "Bug 이슈는 재현 절차가 필수입니다" 에러 표시
        else 재현절차 입력됨
            BL->>A: POST /rest/api/3/issue
        end
    else 이슈 타입 = Story/Task/Epic
        BL->>A: POST /rest/api/3/issue
    end

    A->>WF: 권한 검증 (프로젝트 이슈 생성 권한)
    alt 권한 부족
        WF-->>A: 403 Forbidden
        A-->>BL: 권한 부족 에러
        BL-->>PO: "이슈를 생성할 권한이 없습니다" 메시지 + 이전 화면 유지
    else 권한 있음
        WF-->>A: 허용
        A->>D: 이슈 저장 (상태: Backlog)
        D-->>A: 이슈 키 (PROJ-143)
        A-->>BL: 생성 완료
        BL-->>PO: 백로그에 이슈 추가됨
    end

    Note over PO,D: Sprint Planning 시점
    PO->>BL: 이슈를 스프린트로 드래그
    BL->>A: PUT /rest/api/3/issue/PROJ-143 (sprint_id 설정)
    A->>D: 스프린트 배정 + Audit Log 기록
    A-->>BL: 상태 → Selected for Sprint
```

### 2.3 워크플로우 전환 흐름 (개발 → 코드리뷰 → QA → Done)

```mermaid
sequenceDiagram
    actor DEV as Developer
    actor REV as Reviewer
    actor QA as QA Engineer
    participant BD as 보드 화면(SCR-004/SCR-005)
    participant A as API Server
    participant WF as 워크플로우 엔진
    participant N as 알림 서비스

    DEV->>BD: 이슈 카드를 "In Progress"로 이동
    BD->>A: POST /transitions (→ In Progress)
    A->>WF: 전환 규칙 검증 + WIP 제한 확인
    alt WIP 제한 초과
        WF-->>A: WIP_LIMIT_EXCEEDED (현재 3/3)
        A-->>BD: WIP 제한 초과 경고
        BD-->>DEV: "In Progress 컬럼이 WIP 한도(3)에 도달했습니다.<br/>기존 작업을 먼저 완료하세요." 경고 팝업
        Note over DEV,BD: 팀 정책에 따라 거부 또는 경고 후 진행 선택
    else WIP 허용
        WF-->>A: 허용
        A-->>BD: 상태 변경 완료
    end

    DEV->>BD: PR 생성 후 "Code Review"로 이동
    BD->>A: POST /transitions (→ Code Review)
    A->>WF: 조건 확인 (PR 링크 존재?)
    WF-->>A: 허용
    A->>N: Slack #code-review 알림 + 리뷰어 자동 지정
    A-->>BD: 상태 변경 완료

    REV->>BD: 코드 리뷰 승인
    BD->>A: POST /transitions (→ QA)
    A-->>BD: 상태 변경 완료

    QA->>BD: 테스트 실행
    alt 테스트 통과 - DoD 검증
        QA->>BD: "Done"으로 이동 시도
        BD->>A: POST /transitions (→ Done)
        A->>WF: DoD 체크리스트 검증
        alt DoD 미충족 항목 존재
            WF-->>A: DOD_FAILED [단위테스트 커버리지 80% 미달, 문서 미업데이트]
            A-->>BD: DoD 검증 실패 상세 반환
            BD-->>QA: "Done 전환 불가: 미충족 항목\n- 단위 테스트 커버리지 80% 미달 (현재 65%)\n- API 명세서 미업데이트\n담당자에게 알림을 전송했습니다." 메시지
            A->>N: 담당 개발자에게 DoD 미충족 항목 알림
        else DoD 전체 충족
            WF-->>A: 전체 충족 확인
            A-->>BD: 상태 변경 완료 (Done)
        end
    else 테스트 실패
        QA->>BD: "In Progress"로 반려
        BD->>A: POST /transitions (→ In Progress)
        A->>N: 담당 개발자에게 알림
        A-->>BD: 상태 변경 완료
    end
```

### 2.4 릴리즈 관리 흐름

```mermaid
sequenceDiagram
    actor PM as PM
    participant RL as 릴리즈 화면(SCR-011)
    participant A as API Server
    participant D as Database
    participant N as 알림 서비스

    PM->>RL: 새 버전 생성 (v1.0.0)
    RL->>A: POST /rest/api/3/version
    A->>D: 버전 저장 (Unreleased)
    A-->>RL: 버전 생성 완료

    PM->>RL: 이슈에 Fix Version 지정
    RL->>A: PUT /rest/api/3/issue/{key} (fixVersions)
    A-->>RL: 업데이트 완료

    Note over PM,D: 배포 시점
    PM->>RL: 버전 릴리즈 처리 클릭
    RL->>A: GET /rest/api/3/version/{id}/unresolvedIssueCount
    A->>D: 미완료 이슈 수 조회
    alt 미완료 이슈 존재
        D-->>A: 미완료 이슈 3건 반환
        A-->>RL: 미완료 이슈 경고 데이터
        RL-->>PM: "릴리즈 v1.0.0에 미완료 이슈가 3건 있습니다.\n(PROJ-140, PROJ-141, PROJ-143)\n계속 진행하시겠습니까?" 확인 다이얼로그
        PM->>RL: 확인(진행) 또는 취소 선택
    else 미완료 이슈 없음
        D-->>A: 0건
        A-->>RL: 릴리즈 진행 가능
    end

    PM->>RL: 릴리즈 노트 생성/편집
    RL-->>PM: 릴리즈 노트 에디터 표시<br/>(포함 이슈 자동 목록 + 직접 편집 가능)
    PM->>RL: 릴리즈 노트 내용 작성 후 저장
    RL->>A: PUT /rest/api/3/version (Released + releaseNote)
    A->>D: 상태 변경 + 릴리즈 노트 저장
    A->>N: 릴리즈 완료 알림 발송
    A-->>RL: 릴리즈 완료 (포함 이슈 목록 표시)
```

### 2.5 JQL 검색 흐름

```mermaid
sequenceDiagram
    actor U as 사용자
    participant V as 검색 화면(SCR-008)
    participant A as API Server
    participant P as JQL Parser
    participant D as Database
    participant Cache as 검색 캐시

    U->>V: 검색 바 클릭
    V-->>U: JQL 입력 필드 활성화
    U->>V: JQL 입력 시작 (예: "project = ")
    V->>A: GET /rest/api/3/jql/autocomplete?fieldName=project
    A-->>V: 자동완성 후보 목록 반환
    V-->>U: 드롭다운 자동완성 표시

    U->>V: JQL 완성 후 검색 실행
    V->>A: POST /rest/api/3/issue/search {jql, startAt, maxResults}

    A->>P: JQL 파싱 요청
    P->>P: Lexer (토큰화)
    P->>P: AST 생성 (Abstract Syntax Tree)
    alt JQL 문법 오류
        P-->>A: PARSE_ERROR {position, message}
        A-->>V: 400 Bad Request (JQL 파싱 실패)
        V-->>U: "JQL 오류: 'status' 필드 값 'InProgress'를 찾을 수 없습니다. (line 1, col 12)" 에러 메시지
    else JQL 정상
        P->>P: SQL 변환
        P-->>A: SQL 쿼리 반환
        A->>Cache: 동일 JQL 캐시 확인
        alt 캐시 히트
            Cache-->>A: 캐시된 결과 반환
        else 캐시 미스
            A->>D: SQL 쿼리 실행
            D-->>A: 결과셋 (total, issues[])
            A->>Cache: 결과 캐싱 (TTL 60s)
        end
        A-->>V: 검색 결과 (페이지네이션 포함)<br/>{total: 42, startAt: 0, maxResults: 20, issues: [...]}
        V-->>U: 결과 테이블 표시 + 페이지네이션
    end

    U->>V: 필터 저장 버튼 클릭
    V-->>U: 필터명 입력 다이얼로그
    U->>V: 필터명 입력 후 저장
    V->>A: POST /rest/api/3/filter {name, jql, sharePermissions}
    A->>D: 필터 저장
    A-->>V: 필터 ID 반환
    V-->>U: "필터가 저장되었습니다. 공유 링크: /filters/123"

    U->>V: 필터 공유 버튼 클릭
    V->>A: PUT /rest/api/3/filter/123 (sharePermissions 설정)
    A-->>V: 공유 설정 완료
    V-->>U: 공유 링크 복사 완료 토스트
```

### 2.6 대시보드 가젯 로딩 흐름

```mermaid
sequenceDiagram
    actor U as 사용자
    participant V as 대시보드 화면(SCR-002)
    participant A as API Server
    participant Auth as 권한 서비스
    participant G1 as 가젯:번다운차트
    participant G2 as 가젯:할당된이슈

    U->>V: 대시보드 진입
    V->>A: GET /rest/api/3/dashboard/{id}
    A->>Auth: 사용자 역할 판별 (RBAC)
    Auth-->>A: 역할 반환 (Developer / PM / Admin)
    A->>A: 역할에 따른 가젯 목록 필터링
    A-->>V: 가젯 레이아웃 + 가젯 목록 반환

    V-->>U: 대시보드 레이아웃 표시 (가젯 영역 스켈레톤 UI)

    Note over V,G3: 가젯 비동기 병렬 데이터 fetch
    par 번다운 차트 가젯
        V->>G1: GET /gadget/burndown?sprintId=42
        alt 번다운 데이터 로드 성공
            G1-->>V: 번다운 데이터 반환
            V-->>U: 번다운 차트 렌더링
        else 번다운 가젯 로드 실패
            G1-->>V: 500 Internal Server Error
            V-->>U: 가젯 영역에 "데이터를 불러올 수 없습니다. [새로고침]" 표시
        end
    and 할당된 이슈 가젯
        V->>G2: GET /gadget/assigned-issues?user=currentUser
        alt 이슈 목록 로드 성공
            G2-->>V: 이슈 목록 반환
            V-->>U: 할당 이슈 목록 렌더링
        else 이슈 가젯 로드 실패
            G2-->>V: 503 Service Unavailable
            V-->>U: 가젯 영역에 "서비스 일시 중단. [재시도]" 표시
        end
    and 스프린트 상태 가젯
        V->>G3: GET /gadget/sprint-status?boardId=10
        G3-->>V: 스프린트 진행 현황 반환
        V-->>U: 스프린트 상태 렌더링
    end

    Note over U,V: 가젯 커스터마이징
    U->>V: 가젯 추가 버튼 클릭
    V-->>U: 가젯 카탈로그 다이얼로그 표시
    U->>V: 추가할 가젯 선택
    V->>A: POST /rest/api/3/dashboard/{id}/gadget
    A-->>V: 가젯 추가 완료
    V-->>U: 새 가젯 렌더링

    U->>V: 가젯 드래그로 재배치
    V->>A: PUT /rest/api/3/dashboard/{id}/gadget/{gadgetId} (position 변경)
    A-->>V: 레이아웃 저장 완료
    V-->>U: 재배치된 대시보드 표시

    U->>V: 가젯 제거 버튼 클릭

    A-->>V: 삭제 완료
    V-->>U: 가젯 제거된 대시보드 표시
```

### 2.7 자동화 규칙 실행 흐름

```mermaid
sequenceDiagram
    participant BOARD as 보드 화면(SCR-004/SCR-005)
    participant ES as 이벤트 버스
    participant AE as 자동화 엔진
    participant CE as 조건 평가기
    participant AX as 액션 실행기
    participant SLACK as Slack API
    participant JIRA as Atlassian Jira API
    participant LOG as 실행 로그 DB
    participant DLQ as Dead Letter Queue

    BOARD->>ES: 이벤트 발행 (ISSUE_TRANSITIONED: In Progress → Code Review)
    ES->>AE: 이벤트 전달
    AE->>AE: 이벤트 타입으로 규칙 매칭
    AE-->>AE: 매칭된 규칙 목록 조회<br/>(규칙1: Code Review 전환 시 리뷰어 지정<br/>규칙2: Slack 알림 발송)

    loop 각 매칭 규칙 처리
        AE->>CE: 조건 평가 요청 (issueType, project, assignee 등)
        alt 조건 미충족 (예: Epic 타입은 알림 제외)
            CE-->>AE: CONDITION_NOT_MET
            AE->>LOG: 스킵 로그 기록 (규칙ID, 이슈키, 이유)
            Note over AE: 해당 규칙 스킵, 다음 규칙 진행
        else 조건 충족
            CE-->>AE: CONDITION_MET
            AE->>AX: 액션 실행 요청

            alt 액션: Slack 알림 발송
                AX->>SLACK: POST /chat.postMessage (#code-review 채널)
                alt Slack 발송 성공
                    SLACK-->>AX: 200 OK
                    AX->>LOG: 성공 로그 기록
                else Slack 발송 실패 (재시도 1~3회)
                    SLACK-->>AX: 500 / 429 Too Many Requests
                    AX->>AX: 지수 백오프 재시도 (1s, 2s, 4s)
                    alt 3회 재시도 후 여전히 실패
                        AX->>DLQ: 실패 메시지 DLQ 적재
                        AX->>LOG: 실패 로그 기록 (재시도 횟수, 에러)
                    end
                end
            else 액션: 리뷰어 자동 지정
                AX->>JIRA: PUT /rest/api/3/issue/{key} (reviewer 필드 업데이트)
                JIRA-->>AX: 200 OK
                AX->>LOG: 성공 로그 기록
            else 액션: 댓글 자동 추가
                AX->>JIRA: POST /rest/api/3/issue/{key}/comment
                JIRA-->>AX: 201 Created
                AX->>LOG: 성공 로그 기록
            end
        end
    end

    AE-->>ES: 모든 규칙 처리 완료
```

## 4. 실시간 업데이트 흐름

### 4.1 WebSocket/SSE 기반 실시간 동기화

> **SSE vs WebSocket 선택 기준**
> - **WebSocket**: 클라이언트-서버 양방향 통신이 필요한 경우. 보드 카드 이동(SCR-004/SCR-005), 이슈 상세 댓글 실시간 추가(SCR-006), Planning Poker 세션(SCR-010)에 적용.
> - **SSE (Server-Sent Events)**: 서버→클라이언트 단방향 푸시로 충분한 경우. 대시보드 가젯 자동 새로고침(SCR-002)처럼 주기적 데이터 갱신에 적용. 연결 유지 비용이 낮고 HTTP/2와 호환성이 좋음.

```mermaid
sequenceDiagram
    actor U1 as 사용자A (보드 편집 중)
    actor U2 as 사용자B (같은 보드 열람 중)
    participant V1 as 화면A
    participant V2 as 화면B
    participant WS as WebSocket Server
    participant A as API Server
    participant D as Database

    Note over U1,D: 보드에서 카드 이동 - 실시간 반영 (WebSocket 사용: 양방향 필요)
    V1->>WS: WebSocket 연결 (board:{boardId})
    V2->>WS: WebSocket 연결 (board:{boardId})
    WS-->>V1: 연결 확인
    WS-->>V2: 연결 확인

    U1->>V1: 이슈 카드를 "In Progress" → "Code Review"로 드래그
    V1->>A: POST /transitions {issueKey, transitionId}
    A->>D: 상태 업데이트
    D-->>A: 저장 완료
    A->>WS: BOARD_UPDATED 이벤트 브로드캐스트<br/>{boardId, issueKey, fromStatus, toStatus, updatedBy}
    WS-->>V1: 본인 화면 즉시 반영 (낙관적 업데이트 확정)
    WS-->>V2: 실시간 카드 이동 이벤트 수신
    V2-->>U2: 이슈 카드가 "Code Review" 컬럼으로 자동 이동 표시

    Note over U1,D: 이슈 상세에서 댓글 실시간 추가 (WebSocket 사용: 양방향 필요)
    U1->>V1: 이슈 상세 화면 진입 (SCR-006)
    V2->>WS: WebSocket 연결 (issue:{issueKey})
    U2->>V2: 동일 이슈 상세 화면 열람 중

    U1->>V1: 댓글 입력 후 저장
    V1->>A: POST /rest/api/3/issue/{key}/comment
    A->>D: 댓글 저장
    D-->>A: 저장 완료
    A->>WS: COMMENT_ADDED 이벤트 브로드캐스트<br/>{issueKey, comment, author, createdAt}
    WS-->>V2: 댓글 추가 이벤트 수신
    V2-->>U2: 새 댓글이 댓글 목록에 실시간 추가됨 (페이지 새로고침 불필요)

    Note over U1,D: 대시보드 가젯 자동 새로고침 (SSE 사용: 서버→클라이언트 단방향)
    V2->>WS: SSE 연결 (dashboard:{dashboardId})
    loop 30초마다 자동 새로고침
        WS->>A: 가젯 데이터 갱신 요청 (번다운/스프린트 상태)
        A->>D: 최신 데이터 조회
        D-->>A: 갱신된 데이터
        A->>WS: GADGET_UPDATED 이벤트
        WS-->>V2: SSE 이벤트 전송
        V2-->>U2: 가젯 데이터 자동 갱신 (스피너 없이 부드럽게 업데이트)
    end
```
