## 1. 코드 리뷰 목적

- 코드 품질 향상 및 버그 사전 방지
- 지식 공유 및 팀 역량 강화
- 코딩 표준 준수 확인
- 워크플로우 연동: Code Review 상태에서 리뷰어 승인 시 QA로 전환
- 보안 취약점(SQL Injection, XSS, RBAC 누락) 조기 발견
- 이슈 타입별(Story / Bug / Task) 완료 기준 검증

## 2. 리뷰 프로세스

### 2.1 PR 상태 머신

```mermaid
stateDiagram-v2
    [*] --> DraftPR : PR 초안 생성
    DraftPR --> ReadyForReview : Ready for Review 전환
    ReadyForReview --> InReview : 리뷰어 리뷰 시작
    InReview --> ChangesRequested : Request Changes
    InReview --> Approved : Approve
    ChangesRequested --> Rereview : 작성자 수정 후 재요청
    Rereview --> ChangesRequested : 추가 수정 요청
    Rereview --> Approved : 승인
    Approved --> Merged : 머지
    Merged --> [*]
```

### 2.2 각 단계별 행동 규칙

| 단계 | 행동 주체 | 행동 규칙 |
|------|-----------|-----------|
| Draft PR | 작성자 | 셀프 리뷰 완료 후 Ready for Review로 전환. WIP 코드는 Draft 유지 |
| Ready for Review | 작성자 | PR 설명, 관련 이슈 번호, 스크린샷(UI 변경 시) 첨부 |
| In Review | 리뷰어 | 24시간 이내 첫 코멘트 또는 Approve. 범위 외 지적은 [NICE]로 분류 |
| Changes Requested | 작성자 | 모든 [MUST] 코멘트 해결 후 Re-request review. 반박 시 코멘트로 사유 명시 |
| Re-review | 리뷰어 | 이전 코멘트 해결 여부 중심으로 재검토. 신규 이슈 발견 시 별도 코멘트 |
| Approved | 작성자 | 최소 승인 수 충족 확인 후 Squash Merge 또는 Merge |
| Merged | 작성자 | 관련 이슈 상태 전환 확인(자동 연동 미작동 시 수동 전환) |

### 2.3 프로세스 규칙

| 항목 | 규칙 |
|------|------|
| 리뷰어 수 | 최소 1명 (핵심 변경은 2명) |
| 리뷰 응답 시간 | PR 생성 후 24시간 이내 |
| PR 크기 | 400줄 이하 권장 (초과 시 분할) |
| 셀프 리뷰 | PR 생성 전 필수 |
| 알림 | Slack #code-review 채널 자동 알림 |
| 브랜치 전략 | feature/* → develop → main (GitFlow 기준) |

## 3. 리뷰 체크리스트

> [자동] = CI/봇이 자동 검증 가능 항목 / [수동] = 리뷰어가 직접 확인해야 하는 항목

### 3.1 기능

- [ ] [수동] 요구사항(Acceptance Criteria)을 올바르게 구현했는가?
- [ ] [수동] 엣지 케이스를 처리했는가?
- [ ] [수동] 에러 핸들링이 적절한가?
- [ ] [수동] 워크플로우 전환 규칙을 준수하는가?

### 3.2 코드 품질

- [ ] [수동] 네이밍이 명확하고 일관적인가?
- [ ] [수동] 불필요한 복잡도가 없는가?
- [ ] [자동] 중복 코드가 없는가? (SonarQube Duplication 체크)
- [ ] [수동] 단일 책임 원칙을 따르는가?

### 3.3 보안

- [ ] [자동] SQL Injection 취약점이 없는가? (Snyk / SonarQube)
- [ ] [자동] XSS 취약점이 없는가? (Snyk)
- [ ] [자동] 민감 정보가 하드코딩되지 않았는가? (Gitleaks / SonarQube)
- [ ] [수동] RBAC 권한 체크가 올바른가? (5단계 역할: Admin / Developer / QA / Reporter / Viewer)
- [ ] [수동] 이슈 보안 레벨(Public / Internal / Confidential) 접근 제어가 적절한가?

### 3.4 성능

- [ ] [수동] N+1 쿼리 문제가 없는가?
- [ ] [수동] JQL 검색 쿼리가 최적화되었는가?
- [ ] [수동] 불필요한 API 호출이 없는가?
- [ ] [수동] 적절한 인덱싱이 되어 있는가?

### 3.5 테스트

- [ ] [자동] 새로운 기능에 대한 테스트가 있는가?
- [ ] [자동] 커버리지 80% 이상을 유지하는가? (DoD 조건 — JaCoCo/Istanbul)
- [ ] [수동] 테스트가 의미 있는 케이스를 검증하는가?
- [ ] [자동] 기존 테스트가 깨지지 않았는가? (CI 파이프라인)

### 3.6 이슈 타입별 리뷰 중점

| 이슈 타입 | 리뷰 중점 항목 |
|-----------|----------------|
| **Story** (기능) | Acceptance Criteria 완전 충족 여부, 사용자 시나리오 정상 동작, UI/UX 사양 일치 |
| **Bug** (결함) | 재현 절차대로 버그 재현 확인, 수정 후 동일 경로 재현 불가 확인, 회귀 케이스 추가 여부 |
| **Task** (기술) | 완료 기준(DoD) 항목 체크, 성능/인프라 영향 범위, 문서 업데이트 반영 여부 |

## 4. 리뷰 코멘트 가이드

| 접두사 | 의미 | 예시 |
|--------|------|------|
| [MUST] | 반드시 수정 필요 (머지 블로킹) | [MUST] RBAC 권한 체크가 누락되었습니다 |
| [SHOULD] | 강력 권장 (미수정 시 사유 명시) | [SHOULD] JQL 쿼리에 인덱스를 활용하면 좋겠습니다 |
| [NICE] | 개선 제안 (선택 사항) | [NICE] 변수명을 좀 더 명확하게 바꾸면 좋겠습니다 |
| [Q] | 질문 (코드 이해 목적) | [Q] 이 워크플로우 전환 조건의 의도가 궁금합니다 |

> [MUST] 코멘트가 하나라도 남아 있으면 머지 불가. [SHOULD] / [NICE] 는 작성자 판단으로 처리 가능하나 처리 결과를 코멘트로 남긴다.

### 5.5 Mobile (Flutter/Dart) 코딩 표준

Flutter/Dart 코드는 아래 표준을 준수한다.

#### 린트 도구

| 도구 | 용도 | 설정 방법 |
|------|------|-----------|
| `flutter_lints` | Flutter 공식 권장 lint 규칙 | `pubspec.yaml`에 `flutter_lints` 의존성 추가 |
| `very_good_analysis` | 엄격한 Dart 정적 분석 (선택) | Very Good Ventures 제공, 더 엄격한 규칙 세트 |
| `dart analyze` | Dart 정적 분석 실행 | CI 파이프라인 및 pre-commit hook에 통합 |
| `dart format` | Dart 코드 포매팅 | pre-commit hook에 통합, `dart format .` |

**analysis_options.yaml 설정 예시**

```yaml
include: package:flutter_lints/flutter.yaml

linter:
  rules:
    # 명명 규칙
    - camel_case_types           # 클래스명: UpperCamelCase
    - non_constant_identifier_names  # 변수/함수: lowerCamelCase
    - file_names                 # 파일명: snake_case
    # 코드 품질
    - avoid_print                # print() 사용 금지 (debugPrint 사용)
    - prefer_const_constructors  # 가능한 경우 const 생성자 사용
    - prefer_final_fields        # 변경 없는 필드는 final 선언
    - avoid_unnecessary_containers  # 불필요한 Container 위젯 제거
    # 위젯 구조
    - prefer_const_literals_to_create_immutables
    - sized_box_for_whitespace   # SizedBox 사용 권장 (Container 대신)
```

#### 명명 규칙

| 대상 | 규칙 | 예시 |
|------|------|------|
| 클래스 / 열거형 / typedef | UpperCamelCase | `IssueDetailScreen`, `IssuePriority` |
| 변수 / 함수 / 파라미터 | lowerCamelCase | `issueTitle`, `fetchIssueList()` |
| 파일명 | snake_case | `issue_detail_screen.dart`, `issue_repository.dart` |
| 상수 | lowerCamelCase | `defaultTimeout`, `maxRetryCount` |
| private 멤버 | `_` 접두사 + lowerCamelCase | `_issueList`, `_fetchData()` |

#### 위젯 구조 규칙

| 규칙 | 기준 | 설명 |
|------|------|------|
| 파일당 위젯 수 | 1개 (공개 위젯 기준) | 작은 private 헬퍼 위젯은 같은 파일에 허용 |
| `build()` 메서드 길이 | 200줄 이하 | 초과 시 별도 위젯 또는 메서드로 분리 |
| 상태 분리 | StatelessWidget 우선 | 상태가 필요한 경우만 StatefulWidget 또는 ConsumerWidget 사용 |
| const 생성자 | 가능한 경우 필수 | 불필요한 리빌드 방지 |

#### 상태 관리 규칙 (Riverpod)

| 항목 | 규칙 |
|------|------|
| Provider 네이밍 | `{기능}Provider` (예: `issueListProvider`, `authStateProvider`) |
| 패턴 일관성 | 동일 기능 영역은 동일한 Provider 타입 사용 |
| 비즈니스 로직 위치 | Notifier/StateNotifier에 위치, 위젯에 직접 작성 금지 |
| Provider 범위 | 전역 상태는 최상위 Provider, 화면 로컬 상태는 `AutoDispose` 사용 |

```dart
// 올바른 예: Notifier에 비즈니스 로직 위치
final issueListProvider = AsyncNotifierProvider<IssueListNotifier, List<Issue>>(
  IssueListNotifier.new,
);

class IssueListNotifier extends AsyncNotifier<List<Issue>> {
  @override
  Future<List<Issue>> build() async {
    return ref.read(issueRepositoryProvider).getIssues();
  }

  Future<void> refresh() async {
    state = const AsyncLoading();
    state = await AsyncValue.guard(() =>
        ref.read(issueRepositoryProvider).getIssues());
  }
}
```

## 6. 워크플로우 자동 연동

### 6.1 연동 아키텍처

```mermaid
graph LR
    subgraph GitHub
        PR["Pull Request\n(생성 / Approve / Changes)"]
        GHA["GitHub Actions\nWorkflow"]
    end

    subgraph AtlassianJira
        API["Atlassian Jira Transition API\n/rest/api/3/issue/{key}/transitions"]
        WF["워크플로우 상태\nIn Progress → Code Review\nCode Review → QA\nCode Review → In Progress"]
    end

    PR -->|"이벤트 트리거"| GHA
    GHA -->|"HTTP POST"| API
    API -->|"상태 전환"| WF
    WF -->|"Slack 알림"| Slack["#code-review\nSlack 채널"]
```

### 6.2 전환 규칙

| GitHub 이벤트 | 상태 전환 | 조건 |
|---------------|-----------|------|
| PR 생성 (opened) | In Progress → Code Review | PR 제목 또는 브랜치명에 이슈 키 포함 |
| PR Approve | Code Review → QA | 설정된 최소 Approve 수 충족 |
| PR Request Changes | Code Review → In Progress | 리뷰어가 변경 요청 제출 |
| PR Closed (without merge) | Code Review → In Progress | PR 닫힘(미머지) |
| PR Merged | QA 전환은 별도 처리 | QA 자동화 파이프라인과 연계 |

### 6.3 GitHub Actions YAML 예시 (Transition ID 동적 조회)

Transition ID는 PCH 프로젝트마다 다르므로, GitHub Actions에서 Atlassian Jira REST API(`/rest/api/2/issue/{issueKey}/transitions`)를 호출하여 동적으로 조회하는 것을 권장한다. 하드코딩된 Transition ID는 프로젝트 변경 시 오작동을 유발할 수 있다.

```yaml
# .github/workflows/jira-sync.yml
name: PCH Workflow Sync

on:
  pull_request:
    types: [opened, reopened, closed]
  pull_request_review:
    types: [submitted]

env:
  JIRA_BASE_URL: ${{ secrets.JIRA_BASE_URL }}
  JIRA_USER_EMAIL: ${{ secrets.JIRA_USER_EMAIL }}
  JIRA_API_TOKEN: ${{ secrets.JIRA_API_TOKEN }}

jobs:
  sync-jira:
    runs-on: ubuntu-latest
    steps:
      - name: Extract Issue Key
        id: extract
        run: |
          ISSUE_KEY=$(echo "${{ github.head_ref }}" | grep -oE '[A-Z]+-[0-9]+' | head -1)
          if [ -z "$ISSUE_KEY" ]; then
            ISSUE_KEY=$(echo "${{ github.event.pull_request.title }}" | grep -oE '[A-Z]+-[0-9]+' | head -1)
          fi
          echo "issue_key=$ISSUE_KEY" >> $GITHUB_OUTPUT

      - name: Determine Target Transition Name
        id: target
        run: |
          EVENT="${{ github.event_name }}"
          ACTION="${{ github.event.action }}"
          REVIEW_STATE="${{ github.event.review.state }}"

          if [ "$EVENT" = "pull_request" ] && [ "$ACTION" = "opened" ]; then
            echo "transition_name=Code Review" >> $GITHUB_OUTPUT
          elif [ "$EVENT" = "pull_request_review" ] && [ "$REVIEW_STATE" = "approved" ]; then
            echo "transition_name=QA" >> $GITHUB_OUTPUT
          elif [ "$EVENT" = "pull_request_review" ] && [ "$REVIEW_STATE" = "changes_requested" ]; then
            echo "transition_name=In Progress" >> $GITHUB_OUTPUT
          fi

      - name: Fetch Transition ID Dynamically
        id: transition
        if: steps.extract.outputs.issue_key != '' && steps.target.outputs.transition_name != ''
        run: |
          # Atlassian Jira REST API로 현재 이슈의 가능한 전환 목록 조회 (동적 조회 권장)
          TRANSITIONS=$(curl -s \
            "$JIRA_BASE_URL/rest/api/2/issue/${{ steps.extract.outputs.issue_key }}/transitions" \
            -u "$JIRA_USER_EMAIL:$JIRA_API_TOKEN" \
            -H "Content-Type: application/json")

          TRANSITION_ID=$(echo "$TRANSITIONS" | \
            jq -r --arg name "${{ steps.target.outputs.transition_name }}" \
            '.transitions[] | select(.name == $name) | .id')

          echo "transition_id=$TRANSITION_ID" >> $GITHUB_OUTPUT

      - name: Transition Issue
        if: steps.transition.outputs.transition_id != ''
        run: |
          curl -s -X POST \
            "$JIRA_BASE_URL/rest/api/2/issue/${{ steps.extract.outputs.issue_key }}/transitions" \
            -u "$JIRA_USER_EMAIL:$JIRA_API_TOKEN" \
            -H "Content-Type: application/json" \
            -d "{\"transition\": {\"id\": \"${{ steps.transition.outputs.transition_id }}\"}}"
```

> 위 방식으로 Transition ID를 동적으로 조회하면 프로젝트 설정 변경, 워크플로우 재구성 시에도 YAML 수정 없이 정상 동작한다.

### 7.1 CODEOWNERS 파일 구조

```
# .github/CODEOWNERS

# 전체 코드 — TL이 최소 1명 포함되도록 보장

```

### 7.2 자동 리뷰어 지정 원칙

| 원칙 | 설명 |
|------|------|
| CODEOWNERS 우선 | 변경 파일 경로에 매칭되는 소유자가 1순위 자동 지정 |
| 최근 변경 이력 기반 | `git log --follow` 기준 최근 3개월 내 해당 파일 수정자 추천 |
| 도메인 전문가 포함 | 변경 범위가 2개 이상 도메인에 걸칠 경우 각 도메인 소유자 포함 |
| 라운드로빈 | 동일 팀 내 리뷰 부하 분산 — 직전 리뷰어 제외하고 순환 지정 |

### 7.3 셀프 승인 금지

- 본인이 생성한 PR은 본인이 Approve 불가
- 팀 리더도 본인 PR은 다른 리뷰어의 승인 필요
- 긴급 상황(Hotfix)에서도 최소 1명의 타인 승인 필수

## 8. 긴급 리뷰 프로세스

### 8.1 일반 리뷰 vs 긴급 리뷰 비교

| 항목 | 일반 리뷰 | 긴급 리뷰 (Hotfix) |
|------|-----------|-------------------|
| 리뷰어 수 | 1~2명 | 1명 |
| 응답 SLA | 24시간 | 2시간 |
| PR 크기 제한 | 400줄 이하 | 가능한 최소화 (권장 100줄 이하) |
| 테스트 범위 | 전체 테스트 스위트 | 영향 범위 테스트만 |
| 브랜치 | feature/* → develop | hotfix/* → main + develop |
| 이슈 타입 | Story / Task / Bug | Bug (Priority: Highest) |
| 사후 조치 | 없음 | 정식 리뷰 수행 (72시간 이내) |
| Slack 알림 채널 | #code-review | #code-review-urgent |

### 8.2 긴급 리뷰 발동 기준

- 운영 서비스 장애 발생 (5xx 에러율 급증, 서비스 다운)
- 보안 취약점 즉시 패치 필요
- 데이터 유실 위험이 있는 결함
- 주요 비즈니스 기능(결제, 인증 등) 전면 불가

### 8.3 긴급 리뷰 절차 (브랜치 보호 바이패스 포함)

> 바이패스 권한·필수 CI·Slack `#hotfix-log` 알림 등 **정책 정본**은 [08-Git규칙정의서 5.3절](../08-Git규칙정의서/08-Git규칙정의서_v3.0.md)과 동일하다. (문서검수 Phase 4 교차 참조)

```
1. 작성자: Bug 이슈 생성 (Priority: Highest, Label: hotfix)
2. 작성자: hotfix/PROJ-XXX 브랜치 생성 → 수정 → PR 생성
3. 작성자: PR 제목에 [HOTFIX] 접두사 추가
4. 작성자: Slack #code-review-urgent 에 @on-call 멘션으로 즉시 알림
5. 리뷰어: 2시간 이내 리뷰 완료 → Approve (최소 1명)
6. 머지 담당자 (PM 또는 TL): bypass branch protection 권한으로
   main 브랜치 보호 규칙 우회 후 머지 (1명 승인으로 허용)
7. 작성자: develop 브랜치에도 cherry-pick 또는 merge
8. 자동 알림: Slack #hotfix-log 채널에 머지 완료 알림 자동 발송
9. 사후 의무: 72시간 이내 정식 리뷰 수행 (기술 부채 방지)
```

> main 브랜치 보호 규칙(Required Reviews: 2명)은 PM/TL이 `bypass branch protection` 권한으로 우회 가능하다. 1명 승인 후 머지가 허용되며, 72시간 이내 정식 리뷰는 필수 의무 사항이다. 바이패스 사용 시 Slack #hotfix-log 채널에 자동 알림이 발송되어 이력이 기록된다.

### 8.4 사후 정식 리뷰 의무

| 항목 | 기준 |
|------|------|
| 리뷰 기한 | Hotfix 머지 후 72시간 이내 |
| 리뷰 주체 | 바이패스 시 승인한 PM/TL 외 개발자 1명 이상 |
| 리뷰 방법 | 원본 Hotfix PR 또는 후속 PR에 코멘트로 정식 리뷰 수행 |
| 누락 처리 | 72시간 초과 시 TL이 이슈에 사유 기록 + 다음 스프린트 내 처리 |

## 9. 아키텍처 리뷰 vs 코드 리뷰

### 9.1 구분 기준

| 기준 | 코드 리뷰 | 아키텍처 리뷰 |
|------|-----------|---------------|
| 범위 | 단일 PR / 단일 기능 구현 | 시스템 설계, 모듈 간 관계 |
| 검토자 | 동료 개발자 1~2명 | TL + 관련 도메인 개발자 전원 |
| 소요 시간 | 1~2시간 이내 | 반나절~1일 (회의 포함) |
| 산출물 | PR 코멘트, Approve | ADR(Architecture Decision Record) 문서 |
| 빈도 | 모든 PR | 아키텍처 리뷰 트리거 해당 시만 |
| PCH 연동 | Code Review 상태 | Story/Task + Sub-task(아키텍처 리뷰) 별도 등록 |

### 9.2 아키텍처 리뷰 트리거

아래 조건 중 하나 이상 해당 시 코드 리뷰 전에 아키텍처 리뷰를 먼저 수행한다.

- 새 모듈(패키지/서비스) 추가
- DB 스키마 변경 (테이블 추가/삭제/컬럼 변경)
- API Breaking Change (기존 엔드포인트 제거 또는 요청/응답 스펙 변경)
- 3개 이상의 기존 모듈에 동시 영향
- 외부 시스템 신규 연동 (Atlassian Jira API, Slack, CI/CD 등)
- 인프라 변경 (클라우드 리소스, 네트워크 구성)

## 13. 리뷰 문화 가이드

### 13.1 코드 vs 사람 분리 원칙

- 코멘트의 주어는 항상 코드(또는 구현)이지 사람이 아니다.
- 올바른 예: "이 함수는 단일 책임 원칙을 벗어난 것 같습니다."
- 잘못된 예: "당신은 왜 이렇게 구현했나요?"
- 코드 작성자의 능력이 아니라 코드 자체의 개선 가능성을 논한다.

### 13.2 비동기 리뷰 에티켓

| 규칙 | 설명 |
|------|------|
| 명확한 의도 전달 | 코멘트는 맥락과 이유를 포함한다. "수정 필요" 단독 사용 금지 |
| 즉각 응답 강요 금지 | 리뷰 코멘트에 대한 즉각 답변을 강요하지 않는다 (SLA 내 응답이면 충분) |
| Resolve 전 확인 | 작성자가 코멘트를 Resolve하기 전에 리뷰어 의견이 반영되었는지 확인 |
| 컨텍스트 제공 | 리뷰어는 왜 이 코멘트를 남기는지 배경(보안, 성능, 유지보수)을 함께 설명 |
| 불필요한 핑퐁 방지 | [NICE] 수준 코멘트는 작성자 판단을 존중하고 승인을 블로킹하지 않는다 |

### 13.3 긍정적 피드백도 남기기

- 좋은 구현, 깔끔한 코드, 영리한 해결책을 발견하면 명시적으로 칭찬한다.
- 예: "이 방식으로 N+1 문제를 해결한 것이 인상적입니다."
- 긍정적 피드백은 팀 신뢰 형성과 리뷰 참여 의욕 향상에 직결된다.
- 스프린트 레트로스펙티브에서 좋은 리뷰 사례를 공유하는 문화를 만든다.

### 13.4 질문형 코멘트 권장

- 지적보다 질문으로 시작하면 방어적 반응을 줄일 수 있다.
- 권장 표현:
  - "~하면 어떨까요?"
  - "혹시 ~를 고려해 보셨나요?"
  - "~한 이유가 있을까요? 제가 놓친 맥락이 있을 수 있어서요."
- 질문 후 작성자의 의도가 타당하다면 코멘트를 철회하거나 [NICE]로 낮춘다.
